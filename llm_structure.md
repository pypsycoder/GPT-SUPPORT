# Структура LLM-системы

## Назначение

Этот документ описывает текущее целевое видение LLM-системы целиком.

Это не только схема для `help request`.
Это общая карта того, как система должна обрабатывать разные типы пользовательских сообщений:

- запрос помощи
- логирование / запись данных
- вопрос о контенте
- точечный вопрос
- смешанное сообщение
- потенциально опасное сообщение

Главная идея:

система не должна сразу переходить от сырого сообщения пользователя к `RAG` и генерации ответа.

Сначала она должна понять:

- есть ли safety-риск
- что это за тип сообщения
- одна ли там задача или несколько
- нужен ли вообще `RAG`
- нужен ли `clarifier`
- нужны ли specialists
- что из памяти реально полезно в этом маршруте

## Базовый принцип

Не все сообщения должны проходить через один и тот же pipeline.

Нам нужна многоуровневая orchestration-модель:

1. `safety layer`
2. `memory read layer`
3. `message decomposition / type gate`
4. выбор нужного flow
5. только внутри `help flow` запуск clarifier, retrieval и specialists

Самое важное правило:

`safety` — это не просто первый шаг в списке.
Это глобальный override-слой над всей системой.

Если срабатывает `safety`, обычная orchestration прерывается:

- не запускается обычный `clarifier`
- не запускается обычный `RAG`
- не запускаются обычные specialists
- система сразу уходит в safety-specific route

То есть правильная логика такая:

- сначала `safety screening`
- если `safety positive` -> обычный flow отменяется
- если `safety negative` -> продолжается обычная orchestration

## Верхнеуровневая модель

### Уровень 1. Safety Layer

Это верхний override-слой над всей системой.
Он работает до decomposition, до clarifier, до retrieval и до specialists.

Цель:
как можно раньше отловить сообщения, где нельзя идти в обычный conversational flow.

Что сюда входит:

- суицидальные / кризисные сигналы
- возможные острые состояния
- опасные симптомы
- urgent dialysis-related risk

Если срабатывает `safety layer`, система:

- не уходит в обычный `clarifier`
- не строит длинный retrieval-path
- не делает обычный specialist loop
- переходит в safety-specific response mode

Это абсолютный приоритетный слой.

Важная идея:

`safety` не является обычным flow среди других.
Это условие, которое может прервать любой другой flow.

Система не должна думать так:

- `message -> type gate -> maybe safety`

Система должна думать так:

- `message -> safety screening`
- если опасности нет -> обычная маршрутизация
- если опасность есть -> safety route

### Внутренние типы safety-сценариев

Нам нужны как минимум три safety-ветки:

- `crisis / self-harm`
- `acute medical red flags`
- `urgent dialysis-procedure risk`

#### `crisis / self-harm`

Примеры:

- прямые или косвенные суицидальные высказывания
- ощущение, что человек может причинить себе вред

Ответ:

- кризисный safety-response
- минимально необходимые инструкции
- немедленная эскалация в crisis-mode

#### `acute medical red flags`

Примеры:

- потенциально опасные симптомы
- выраженное ухудшение самочувствия
- признаки состояния, которое нельзя разбирать как обычный support-case

Ответ:

- короткий и ясный triage-style response
- минимально необходимая медицинская осторожность
- без длинного conversational exploration

#### `urgent dialysis-procedure risk`

Примеры:

- риск, связанный с текущим или ближайшим сеансом диализа
- симптомы во время процедуры
- ситуация, требующая быстрого контакта с персоналом

Ответ:

- быстрый route в dialysis-risk response
- четкая практическая инструкция без лишней терапии разговором

### Что safety-слой может и чего не может

Safety-слой может:

- прервать обычную orchestration
- ограничить число вопросов
- запретить обычные CTA
- запретить обычный specialist loop

Safety-слой не должен:

- уходить в длинный help-clarifier
- задавать открытые exploratory-вопросы до базового triage
- запускать обычный educational flow

## Memory Layer

Память должна быть отдельным слоем системы, а не случайной смесью из `chat_history`, patient summary и retrieval.

Нам нужны два разных типа памяти:

- `LT-memory` — долгосрочная память о пациенте
- `ST-memory` — краткосрочная память текущей сессии / текущей активной ветки

Они нужны для разных задач и не должны смешиваться.

### 1. LT-memory

`LT-memory` хранит относительно устойчивые сведения о пациенте, которые полезны в разных сессиях.

Это не полный медицинский архив и не сырая история всех диалогов.
Это curated memory для маршрутизации, персонализации и выбора формы помощи.

Что стоит туда писать:

- устойчивые предпочтения в стиле помощи
- повторяющиеся паттерны проблем
- повторяющиеся триггеры
- предпочтение уроков или практик
- склонность лучше откликаться на `routine`, `psych_support` или `education`
- значимый устойчивый прогресс по урокам и практикам

Примеры:

- лучше откликается на короткие practical steps
- часто пишет про тревогу перед диализом
- не любит длинные объяснения
- уже хорошо освоил конкретный lesson cluster

Что не стоит туда писать:

- каждую эмоцию одного дня
- случайные детали одного разговора
- сырые формулировки модели
- сомнительные догадки
- быстро устаревающие детали

Правило:

в `LT-memory` попадает только то, что либо повторяется, либо реально меняет будущую помощь.

### 2. ST-memory

`ST-memory` хранит рабочий контекст текущей сессии или текущей активной ветки.

Это память не про "кто пациент вообще", а про "что сейчас происходит в этом разговоре".

Что стоит туда писать:

- текущую тему разговора
- уже выделенные `problem / intent / context`
- уже заданные уточняющие вопросы
- ответы на них
- текущий активный flow
- текущий help mode
- явные ограничения от пользователя
- незавершенные подзадачи

Примеры:

- сейчас обсуждаем плохой сон и разбитость сегодня
- уже уточнили, что это не после диализа
- пользователь не хочет урок, а хочет помощь пережить день
- сейчас активен `routine-first` mode

Что не стоит туда писать:

- всю переписку подряд без отбора
- детали, которые уже не влияют на текущую ветку
- шум, не меняющий маршрут

Правило:

`ST-memory` должна помогать продолжать текущий разговор, а не тащить за собой весь его текст.

### 3. Когда читать LT-memory

`LT-memory` нужно читать:

- после `safety`, но до окончательного route selection
- перед выбором формы помощи
- перед content recommendation
- перед final CTA choice

Особенно полезно читать `LT-memory`, когда:

- запрос короткий и недоопределенный
- допустимо несколько нормальных ответов
- важно не повторить то, что пациенту уже не подходит

`LT-memory` не должна:

- подменять текущий запрос
- перетягивать разговор в старую тему
- доминировать над свежим контекстом

### 4. Когда читать ST-memory

`ST-memory` нужно читать:

- сразу после `safety`
- до `message decomposition`
- до `clarifier`
- до обработки коротких follow-up сообщений
- до повторного retrieval в рамках той же ветки

Особенно важна `ST-memory` для:

- multi-turn clarification
- follow-up сообщений вида «да», «нет», «именно после диализа»
- mixed flow
- удержания активной задачи разговора

### 5. Когда писать в память

#### Запись в ST-memory

Писать после каждого meaningful шага:

- после decomposition
- после clarifier question
- после ответа пользователя на clarifier
- после выбора help mode
- после route selection
- после разруливания mixed-message

#### Запись в LT-memory

Писать редко и только после отбора.

Хорошие поводы:

- паттерн повторился несколько раз
- пользователь явно выразил устойчивое предпочтение
- видно устойчивое избегание какого-то формата помощи
- завершен значимый цикл обучения / практики

### 6. Что память не должна делать

Память не должна:

- заменять factual patient data
- заменять `chat_history`
- заменять retriever
- становиться свалкой автоматически сгенерированных summary

Memory — это не "все, что система знает".
Memory — это отобранный слой, который помогает маршрутизации и персонализации.

### 7. Кто решает, что писать в память

Ключевое правило:

ни specialists, ни composer, ни финальный ответ не должны писать память напрямую.

Память должна заполняться через отдельный `memory writer / memory gate`.

Логика такая:

1. разные части системы создают `memory candidates`
2. `memory writer` фильтрует их по policy
3. только approved entries записываются в `ST-memory` или `LT-memory`

То есть вопрос "что писать" решает не один агент, а связка:

- `candidate producer`
- `memory writer policy`
- `memory store`

### 8. Матрица ответственности

#### Clarifier

Что может предложить как `memory candidate`:

- текущую главную проблему
- текущий intent
- важный context-fact из уточнения
- предпочтение, явно сказанное пользователем в этой сессии

Что может писать:

- только в `ST-memory`

Что не должен писать:

- устойчивые выводы о личности пациента
- долгосрочные предпочтения на основе одного диалога

#### Message Router / Orchestrator

Что может предложить как `memory candidate`:

- активный flow
- активную ветку разговора
- незавершенную подзадачу
- текущий help mode

Что может писать:

- только в `ST-memory`

Что не должен писать:

- долгосрочные behavioral conclusions
- устойчивые preferences без подтверждения

#### Progress / Education Layer

Что может предложить как `memory candidate`:

- lesson completed
- lesson passed
- linked practice completed
- устойчивый learning milestone

Что может писать:

- в `ST-memory` как текущий контекст
- в `LT-memory` как устойчивый progress fact

Что не должен писать:

- субъективные выводы о мотивации пользователя

#### Content / CTA Selector

Что может предложить как `memory candidate`:

- пользователь явно выбрал урок
- пользователь явно выбрал практику
- пользователь явно отказался от контента в этой сессии

Что может писать:

- в `ST-memory`

Иногда может писать в `LT-memory`, только если:

- это повторяющееся и устойчивое preference signal

#### Specialists

Что могут делать:

- использовать уже переданный context

Что не могут делать:

- напрямую писать в `ST-memory`
- напрямую писать в `LT-memory`
- формировать долгосрочные выводы о пациенте

Причина:

specialists слишком локальны и слишком зависят от случайного текущего retrieval.

#### Composer

Что может делать:

- использовать уже собранные outputs

Что не может делать:

- писать память
- делать выводы о долгосрочных предпочтениях

#### Critic / Rewrite

Что может предложить как `memory candidate`:

- почти ничего

Исключение:

- иногда может поставить технический diagnostic flag, но это скорее system telemetry, а не patient memory

### 9. Что можно писать в ST-memory

В `ST-memory` можно писать:

- `current_problem`
- `current_intent`
- `current_context_fact`
- `active_flow`
- `active_help_mode`
- `clarifier_question_asked`
- `clarifier_answer`
- `open_subtask`
- `user_constraint_for_this_session`

Примеры:

- `current_problem = low_energy_today`
- `current_intent = practical_day_support`
- `clarifier_answer = not_after_dialysis`
- `user_constraint_for_this_session = no_lesson_now`

### 10. Что можно писать в LT-memory

В `LT-memory` можно писать только:

- устойчивое preference
- устойчивый repeated pattern
- устойчивый trigger
- проверенный progress milestone
- подтвержденную format preference

Примеры:

- предпочитает короткие practical ответы
- часто пишет про тревогу перед диализом
- чаще выбирает практики, чем уроки
- лучше откликается на routine-first support

### 11. Что нельзя писать вообще

Нельзя писать ни в `ST`, ни в `LT`:

- сырые тексты модели как есть
- разовые эмоции без устойчивого значения
- догадки о характере пользователя
- случайные интерпретации retrieval
- выводы на основе одного specialist response
- медицинские выводы, которых система не имеет права делать

Примеры плохих записей:

- "пользователь тревожный человек"
- "не любит дыхание"
- "ему неинтересно обучение" после одного отказа
- "с ним лучше всегда psych_support"

### 12. Правило записи в LT-memory

Запись в `LT-memory` разрешена, только если выполняется хотя бы одно из условий:

- это явно сказано пользователем как устойчивое preference
- это подтвердилось несколько раз
- это пришло из надежного progress event
- это заметно меняет будущий routing или content choice

Если условие не выполнено:

- запись идет только в `ST-memory`
- или не пишется вообще

### 13. Практическое правило по умолчанию

Если есть сомнение:

- писать в `ST-memory`
- не писать в `LT-memory`

То есть система по умолчанию должна быть консервативной:

- `ST` писать можно чаще
- `LT` писать редко
- specialists не пишут память вообще

### 14. Практическая policy-таблица записи памяти

Ниже — рабочая таблица для нашего проекта.

#### Таблица: событие -> куда писать -> кто решает

| Событие | Писать в ST | Писать в LT | Кто решает | Правило |
|---|---|---|---|---|
| Определен `message_type` | Да | Нет | router / memory writer | Нужен для текущего route, но не для долгой памяти |
| Выделена главная проблема | Да | Нет | clarifier / memory writer | Это рабочий контекст текущей ветки |
| Определен текущий `intent` | Да | Нет | clarifier / memory writer | Это сессионная цель, а не устойчивое свойство |
| Получен ответ на clarifier | Да | Нет | clarifier / memory writer | Сохраняем до конца ветки |
| Выбран `help_mode` | Да | Нет | orchestrator / memory writer | Нужен для продолжения текущего диалога |
| Пользователь сказал “не хочу урок сейчас” | Да | Нет | clarifier / memory writer | Это ограничение текущей сессии, пока не повторилось |
| Пользователь несколько раз в разных сессиях избегает уроков | Да | Да | LT memory writer | Писать в LT только после повторяемости |
| Пользователь явно сказал “лучше коротко и по делу” | Да | Да | LT memory writer | Явное устойчивое preference можно писать в LT |
| Пользователь один раз сказал “сейчас просто поддержи” | Да | Нет | clarifier / memory writer | Одного раза мало для LT |
| Повторяющийся паттерн “тревога перед диализом” | Да | Да | LT memory writer | Нужны повторные подтверждения |
| Один эпизод тревоги / усталости | Да | Нет | memory writer | Разовая эмоция не идет в LT |
| Lesson completed / passed | Да | Да | progress layer / LT memory writer | Это устойчивый progress fact |
| Practice completed один раз | Да | Нет | progress layer / memory writer | Один completion — еще не LT-паттерн |
| Practice chosen repeatedly | Да | Да | LT memory writer | Можно писать как preference toward practice |
| Router выбрал `routine-first` один раз | Да | Нет | orchestrator / memory writer | Это решение маршрута, а не устойчивое свойство |
| Router стабильно выбирает один и тот же mode для похожих кейсов | Да | Да | LT memory writer | Только после повторения и review policy |
| Specialist сгенерировал advice | Нет | Нет | никто | Specialist output не пишет память напрямую |
| Composer собрал ответ | Нет | Нет | никто | Composer не пишет память |
| Critic нашел ошибку | Нет | Нет | telemetry only | Это system telemetry, не patient memory |

#### Правила для LT-перехода

Запись в `LT-memory` разрешена только если выполнено одно из правил:

1. `explicit_user_preference`
   Пользователь сам явно формулирует устойчивое предпочтение.

2. `repeated_pattern`
   Один и тот же паттерн подтверждается несколько раз в разных сессиях.

3. `progress_event`
   Есть надежное событие прогресса из education / practice layer.

4. `stable_behavior_signal`
   Поведение устойчиво влияет на routing или content choice и повторилось достаточно раз.

#### Рекомендуемые пороги

Для первой рабочей версии можно брать такие conservative thresholds:

- `explicit_user_preference` -> можно писать в LT сразу
- `repeated_pattern` -> минимум `2-3` независимых подтверждения
- `stable_behavior_signal` -> минимум `3` похожих эпизода
- `progress_event` -> писать сразу, если это factual completion / passed result

#### TTL и срок жизни

`ST-memory`:

- живет в рамках текущей сессии или активной ветки
- может иметь TTL от нескольких минут до конца сессии
- очищается или архивируется после завершения ветки

`LT-memory`:

- живет долго
- но должна поддерживать review / decay / overwrite policy
- более новые и более устойчивые сигналы должны уметь ослаблять старые

#### Что делать, если не уверены

Если система не уверена:

- записать в `ST-memory`
- не записывать в `LT-memory`

Это основное защитное правило против засорения долгосрочной памяти.

## Message Decomposition / Type Gate

### Уровень 2. Message Decomposition / Type Gate

Цель:
понять, что это за сообщение и состоит ли оно из одной или нескольких задач.

Система должна уметь:

- определить тип сообщения
- разложить mixed-message на подзадачи
- понять, нужен ли LLM-heavy flow вообще

Типы сообщений:

- `help_request`
- `tracking_request`
- `content_request`
- `specific_question`
- `administrative_or_simple_command`
- `mixed`

Примеры:

- `help_request`
  «Мне тревожно перед диализом, помоги»

- `tracking_request`
  «Запиши давление 150 на 90»

- `content_request`
  «Что почитать про сон?»

- `specific_question`
  «Можно съесть арбуз?»

- `mixed`
  «Запиши давление и скажи, нормально ли это»

Важно:

`message type gate` — это уже часть оркестратора.
Это первый слой обычной маршрутизации после `safety` и memory-read.

### Уровень 3. Flow Selection

После `type gate` система выбирает один из потоков:

- `safety flow`
- `action / tracking flow`
- `content flow`
- `specific question flow`
- `help flow`
- `mixed flow`

Не все сообщения должны доходить до `help flow`.
Не все сообщения должны идти в `RAG`.
Не все сообщения требуют specialists.

## Полная карта потоков

### 1. Safety Flow

Когда нужен:

- crisis / self-harm
- acute symptoms
- clinical red flags

Что делает:

- быстрый triage
- safety-oriented response
- минимально необходимая эскалация
- при необходимости срочный route

Чего не делает:

- не задает лишние открытые вопросы
- не уходит в длинное уточнение intent
- не строит обычный educational CTA

Дополнение:

`safety flow` не равен обычному flow selection.
Он может быть активирован как прерывающий режим поверх остальных сценариев.

### 2. Action / Tracking Flow

Когда нужен:

- пользователь просит записать показатель
- отметить событие
- сохранить действие
- показать уже сохраненные данные

Примеры:

- «Запиши мое давление»
- «Отметь сон»
- «Запиши, что я сделал практику»

Что делает:

- выполняет действие
- подтверждает результат
- при необходимости добавляет короткое пояснение

Что обычно не нужно:

- `clarifier`
- specialists
- `RAG`

Иногда возможен hybrid-переход:

- «Запиши давление 150 на 90 и скажи, нормально ли это»

Тогда сообщение сначала декомпозируется на:

- action subtask
- question subtask

### 3. Content Flow

Когда нужен:

- пользователь явно хочет материал
- урок
- практику
- подбор контента по теме

Примеры:

- «Что почитать про сон?»
- «Какая практика есть на тревогу?»
- «Дай урок про стресс»

Что делает:

- выбирает релевантный контент
- учитывает progress и `LT-memory`
- предлагает lesson или practice

Что обычно не нужно:

- emotional specialists
- routine specialists
- full `clarifier`

Иногда можно добавить короткое уточнение:

- «Тебе сейчас больше нужен урок или короткая практика?»

Но только если это реально меняет подбор.

### 4. Specific Question Flow

Когда нужен:

- есть конкретный предметный вопрос
- пользователь хочет ответ, а не support flow

Примеры:

- «Можно съесть арбуз?»
- «Почему после диализа кружится голова?»
- «Это нормально после процедуры?»

Что делает:

- определяет тип вопроса
- при необходимости делает targeted retrieval
- отвечает кратко и по делу
- при необходимости добавляет cautious guidance

Что не нужно по умолчанию:

- `clarifier` как для help-flow
- specialists loop

Но если вопрос на самом деле скрыто mixed:

- «Можно съесть арбуз? Я вообще в последнее время тревожусь и не понимаю, что мне можно»

тогда `type gate` должен заметить смешанную структуру.

### 5. Help Flow

Это самый сложный conversational flow.
Именно здесь нужен `clarifier`, targeted `RAG` и specialists.

Help flow нужен, когда пользователь пишет за помощью, а не только за фактом, записью или контентом.

Примеры:

- «Мне тяжело после плохой ночи, что делать?»
- «Тревожно перед диализом, помоги»
- «Не понимаю, как сегодня прожить день»

### 6. Mixed Flow

Когда нужен:

- сообщение содержит несколько разных задач

Примеры:

- «Запиши давление и скажи, нормально ли это»
- «Я плохо спал, и кстати что почитать про сон?»
- «Можно съесть арбуз? И еще мне тревожно перед диализом»

Что делает:

- раскладывает сообщение на подзадачи
- определяет, какая задача главная
- решает, обрабатывать ли их вместе или по очереди

Возможные стратегии:

- выполнить сначала action, потом ответить на вопрос
- сначала ответить на urgent-question, потом на вторую часть
- для `help + content` дать основной ответ и мягко добавить контент

## Help Flow: внутренняя структура

### 1. Help Clarifier

Цель:
понять, достаточно ли у системы информации для ответа.

Главная идея:

`clarifier` не должен задавать вопрос всегда.
Он должен сначала определить, хватает ли данных.

Если данных уже достаточно, вопросы не нужны.
Если данных недостаточно, он может задать не один, а несколько вопросов, но только пока это действительно улучшает выбор следующего шага.

Критерий остановки:

не «задать один вопрос»,
а «информации достаточно, чтобы выбрать help mode и retrieval path».

### 2. Что clarifier должен понять

Для `help_request` ему нужно собрать минимум по трем осям:

- `problem`
  с чем столкнулся пользователь
- `intent`
  зачем он пишет именно сейчас
- `context`
  в какой ситуации развилась проблема

#### Problem

Что случилось.
Если проблем несколько, нужно выбрать главную.

Примеры:

- плохой сон ночью
- разбитость сегодня
- тревога перед диализом
- слабость после процедуры

#### Intent

Какой тип помощи пользователь хочет сейчас.

Примеры:

- посочувствовать / поддержать
- помочь пережить день
- помочь со сном
- объяснить, что происходит

Это критично, потому что от intent зависит, нужны ли `psych_support`, `routine`, `education` — или некоторые из них вообще не нужны.

#### Context

В какой ситуации развилась проблема.

Примеры:

- после диализа или нет
- разово или повторяется
- больше эмоция или физическое состояние
- проблема про сейчас или в целом

### 3. Типы уточняющих вопросов

Есть два основных типа уточнений.

#### Открытый вопрос про цель помощи

Нужен, когда неясно, чего человек хочет в ответ.

Пример:

- «Что тебе сейчас важнее: немного поддержки, помощь пережить день или советы именно про сон?»

#### Уточняющий вопрос, который меняет advice

Нужен, когда одна деталь меняет retrieval или specialist routing.

Пример:

- «Это после диализа или просто была плохая ночь?»

### 4. Когда clarifier не нужен

Clarifier не нужен:

- если это не `help_request`
- если это help-request, но и так уже ясно:
  - главная проблема
  - намерение пользователя
  - контекст

Пример:

- «Мне тревожно перед диализом, помоги успокоиться»

Тут, скорее всего, достаточно информации сразу.

## Retrieval внутри Help Flow

### 1. Почему нельзя идти в RAG слишком рано

Если идти в retrieval по сырому сообщению, система легко смешивает разные сущности:

- причина
- текущее состояние
- цель помощи

Пример:

- «плохо спал ночью»
- «сегодня разбит»
- «что делать сегодня»

Если это превращается в один embedding query, retrieval начинает приносить средне-релевантные chunk-и вместо точных.

### 2. Targeted RAG

После clarifier система должна строить retrieval не по raw message, а по issue-aware structure.

Пример:

- отдельный запрос под `sleep_problem`
- отдельный запрос под `low_energy_today`
- при необходимости отдельный запрос под `emotional_coping`

Это позволяет не тащить случайные chunk-и только из-за слов вроде «усталость».

### 3. Role-aware chunk filtering

Даже после retrieval нельзя всем specialist-агентам давать одинаковый набор chunk-ов без фильтрации.

#### `psych_support`

Предпочитает:

- emotion-related practice
- anxiety / stress / emotional coping chunk-и

Избегает:

- физической активности как основной опоры
- chunk-ов про усталость без эмоционального содержания

#### `routine`

Предпочитает:

- short-horizon practical chunk-и
- pacing, снижение нагрузки, маленькие шаги

Избегает:

- content recommendation
- explanation-only chunk-ов

#### `education`

Предпочитает:

- explanatory chunk-и
- lesson-linked content
- chunk-и, которые поддерживают lesson / practice CTA

## Specialists

Specialists нужны не всегда.
Они нужны в help-flow, когда уже понятно, что нужен именно support response.

### `psych_support`

Роль:

- признать текущее эмоциональное состояние
- предложить один grounded способ помочь себе сейчас

Не его роль:

- вести в урок
- тащить CTA
- объяснять весь механизм
- давать длинный список действий

### `routine`

Роль:

- дать 1-2 шага на ближайшие часы
- опционально одну мягкую дневную опору

Не его роль:

- давать lesson CTA
- подменять ответ «почитайте урок»
- быть educational-agent

### `education`

Роль:

- дать короткое объяснение по chunk-у
- добавить мягкий progress-aware CTA

CTA policy:

- lesson, если он не пройден или не усвоен
- practice, если lesson пройден и есть linked practice

Не его роль:

- быть главным emotional support agent
- быть главным practical step agent

## Composer

Цель:
собрать один итоговый ответ из нужных специалистов, не придумывая новых смыслов.

Порядок ответа:

1. короткое признание состояния
2. 1-2 уместных next steps
3. объяснение, только если оно реально нужно
4. мягкий CTA, только если он не заменяет сам ответ

Важно:

- не все specialists должны участвовать всегда
- иногда `education` или `routine` действительно не нужны

## Critic / Rewrite

Цель:
ловить ошибки качества и маршрута после генерации.

Что должен проверять critic:

- unsupported advice
- topic drift
- generic reassurance
- преждевременную или лишнюю эскалацию
- CTA вместо настоящей помощи
- role collapse между specialists

Rewrite должен исправлять:

- шаблонные утешения
- плохую эскалацию
- advice вне grounding
- слишком общие и неестественные ответы

## Cross-turn state

Это отдельный обязательный слой, который пока легко недооценить.

Система должна помнить:

- какой вопрос мы уже задали
- на что пользователь сейчас отвечает
- какая ветка разговора активна
- есть ли незавершенное уточнение

Примеры:

- «да, именно после диализа»
- «нет, я не про сон, а про день»
- «не хочу урок, просто помоги пережить сегодня»

Без cross-turn state система будет ошибочно заново классифицировать такие сообщения как самостоятельные.

Важно:

`cross-turn state` ближе к `ST-memory`, чем к `LT-memory`.
По сути это operational subset краткосрочной памяти, который нужен для удержания текущей ветки.

## Диагностика и eval

Нужно оценивать не только финальный ответ, но и весь маршрут.

Важные элементы trace:

- safety decision
- LT-memory reads
- ST-memory reads
- message type decision
- decomposition результата
- нужен ли clarifier
- какие вопросы были заданы
- когда система решила, что информации достаточно
- retrieval queries
- выбранные chunk-и
- role-aware filtering
- specialist outputs
- composer decision
- critic verdict
- memory writes

Полезные метрики:

- wrong-route rate
- unnecessary-clarifier rate
- missing-clarifier rate
- stale-memory-usage rate
- useless-memory-write rate
- wrong-topic chunk rate
- unsupported advice rate
- CTA correctness
- rewrite frequency

## Итоговая модель

Полная целевая схема выглядит так:

```text
сообщение пользователя
-> safety screening layer
   -> if safety positive: safety flow
   -> if safety negative: continue
-> read ST-memory
-> read LT-memory
-> message decomposition / type gate
-> flow selection
   -> action / tracking flow
   -> content flow
   -> specific question flow
   -> help flow
   -> mixed flow

help flow:
-> clarifier
-> information sufficiency check
-> targeted retrieval query plan
-> RAG retrieval
-> role-aware chunk filter
-> specialists
   -> psych_support
   -> routine
   -> education
-> composer
-> critic / rewrite

all flows:
-> selective memory write
-> diagnostics / trace / eval
```

```text
message

-> compose and check the answer
```

## Главное изменение мышления

Система не должна думать так:

- «любое сообщение -> RAG -> answer»

Она должна думать так:

- «есть ли здесь риск?»
- «что это за сообщение?»
- «одна тут задача или несколько?»
- «что из памяти реально полезно именно здесь?»
- «нужен ли вообще help-flow?»
- «если это help-flow, достаточно ли информации?»
- «если информации мало, какой следующий вопрос реально изменит путь?»
- «только теперь идти в retrieval и specialists»

Это и есть текущее целевое видение оркестрации.
