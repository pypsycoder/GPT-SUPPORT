Карта оркестрации: кто что делает, что получает и что передаёт дальше

Эта схема собрана по текущему коду в app/llm/agent.py, app/llm/context_builder.py, app/llm/orchestration.py и orchestration-prompts. Она показывает основной путь llm_full, ранние выходы и ключевые пакеты данных между слоями.

Главная идея

  • Сначала система собирает контекст пациента, память, intake и role-specific RAG views.
  • Потом router выбирает не более двух specialist-веток и назначает primary-agent.
  • Каждый specialist пишет строго структурированный JSON-вклад: что увидел, что советует, чего избегать.
  • Composer собирает единый ответ только из specialist outputs, а critic проверяет согласованность и policy.
  • Параллельно пишутся diagnostics, memory candidates и shadow validation.

Легенда

CONTROL управляет маршрутом CONTEXT собирает и фильтрует данные OUTPUT формирует вклад или ответ GUARD проверяет, режет или останавливает

2. Основная orchestration-ветка

Это путь для orchestration_mode = llm_full. Он начинается после context-builder и заканчивается composer/critic-парой.

llm_full path

Router Specialist Selector

route decision

LLM-router выбирает до двух specialist-агентов и назначает primary/secondary. После этого route дополнительно постобрабатывается кодом.

  • selected_agents
  • primary_agent
  • secondary_agents
  • routing_reasons, why_not_selected
Постпроцесс route
  • routine выкидывается, если rag_views["routine"] пуст.
  • Если primary был routine, он переключается на оставшегося агента.
↓
parallel specialist stage

psych_support

practice-only specialist

Видит только practice metadata и role-specific psych-support view. Не имеет права выдумывать технику, если grounded practice нет.

  • input: patient_summary_prompt, practice_metadata
  • output: signals
  • output: recommended_actions
  • output: draft, notes_for_composer

routine

short-horizon practical specialist

Берёт только actionable fragments из rag_view_routine. Не предлагает уроки и не пишет бытовые советы из головы.

  • input: patient_summary_prompt, rag_view_routine
  • output: 1–2 short-horizon actions
  • output: neutral draft if RAG empty

education

short explanatory specialist

Использует только general rag_context. Может добавить мягкий CTA из cta_*, если он подтверждён retrieval metadata.

  • input: patient_summary_prompt, rag_context
  • output: draft
  • output: cta_type, cta_label, cta_target
↓ specialists[]

Composer

single final draft

Собирает единый ответ пациенту. Может использовать только то, что уже пришло из specialist outputs: draft, recommended_actions, cta_*, notes_for_composer.

  • output: draft_response
  • output: blocks
  • output: composition_rules
  • output: guidance_text
Что ему запрещено
  • Новые техники, sleep-hygiene и эскалация без specialist grounding.
  • Action block, если у specialists пустой recommended_actions.
↓

Critic

quality + policy guard

Проверяет route/response alignment, early escalation, ungrounded technique, template reassurance и необходимость rewrite.

  • status: pass / fail
  • violations
  • rewrite_required
  • rewrite_reasons
↓

Final Response + Shadow Validation

visible answer + diagnostics

Если critic пропускает, пациент получает composer draft. Параллельно legacy validator проверяет те же риски и сравнивается с critic verdict.

  • patient-visible response
  • shadow_validation
  • matches_critic
  • rewrite.status

3. Ранние выходы и guard-ветки

Эти маршруты могут остановить обычный путь до specialist loop или вместо него.

early exits

Safety Route

highest priority override

Для crisis / acute medical / urgent dialysis-risk. Обычная orchestration не запускается.

  • нет обычного specialist loop
  • нет обычного retrieval route
  • ответ идёт по safety-specific mode

Clarifier Exit

help request but not enough info

Если intake видит help-request, но данных не хватает, система возвращает уточняющий вопрос и не идёт в main LLM response path.

  • clarification_needed = true
  • rewrite.status = skipped_clarifier

Boundary Guard

before LLM call

Останавливает ответ, если пользователь пытается выйти за допустимую границу поведения системы.

  • LLM call skipped
  • готовый boundary response
  • diagnostics stay intact

Ключи

Кто за что отвечает

  • Context Builder собирает факты и role-specific retrieval views.
  • Router решает, какие специалисты вообще должны участвовать.
  • Specialists дают узкие, grounded-вклады в JSON-формате.
  • Composer собирает пациентский ответ, ничего не придумывая поверх specialist outputs.
  • Critic проверяет policy, качество и согласованность route ↔ response.

Самые полезные поля в trace

  • patient_context.rag.views.*_count — какие ветки реально получили grounded контекст.
  • orchestration.route — кого выбрал router и кого потом срезал postprocess.
  • specialists[*].recommended_actions — есть ли grounded action вообще.
  • composer.normalized_output — что реально ушло в финальный draft после guard-логики.
  • shadow_validation — совпадает ли legacy validator с critic.

Как читать проблемный кейс

Если ответ выглядит странно, обычно удобно идти справа налево:

  • Сначала посмотреть финальный composer.draft_response.
  • Потом проверить, что реально дали specialists.
  • Потом проверить, были ли у них вообще нужные rag_views.
  • И только потом смотреть, правильно ли router выбрал route.
