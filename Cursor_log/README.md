# 📋 Правило логирования Cursor

## О правиле

Это правило автоматически напоминает о необходимости создавать лог-отчёты после завершения работы в Cursor.

**Статус:** ✅ Активно
**Область применения:** Всегда (alwaysApply: true)
**Файл:** `.cursor/rules/agent-logging.mdc`

---

## 🎯 Цель

- Создавать историю работы проекта
- Быстро находить, что было сделано
- Отслеживать изменения и проблемы
- Документировать процесс миграций

---

## 📝 Как использовать

### После завершения задачи:

1. **Создайте файл** в папке `Cursor_log/`
2. **Используйте формат:** `YYYY-MM-DD_HH-MM_НазваниеЗадачи.md`
3. **Заполните отчёт** согласно шаблону

### Шаблон отчёта

```markdown
**Задача:** [описание]
**Дата:** [дата время]
**Действия:** [список]
**Файлы:** [список]
**Результат:** [итог]
**Проблемы:** [если есть]
```

### Пример

```markdown
**Задача:** Проверка сохранения PSQI в БД

**Дата:** 2026-02-07 20:58

**Действия:**
- Проанализирован код PSQI
- Найдена проблема с UUID
- Подготовлена миграция

**Файлы:**
- `alembic/versions/3d3b0b2c7bf8_add_scale_results_table.py` (изменён)

**Результат:** Проблема выявлена и исправлена.

**Проблемы:** Нет
```

---

## ⏱️ Когда создавать отчёты

✅ Создавайте после:
- Завершения основной задачи
- Создания резервных копий
- Применения миграций
- Рефакторинга
- Создания функционала
- Решения проблем

## 📊 Управление логами

### Просмотр всех отчётов

```bash
ls Cursor_log/
# или
dir Cursor_log
```

### Поиск отчёта по дате

```bash
ls Cursor_log/2026-02-07_*
```

### Поиск отчёта по названию

```bash
ls Cursor_log/*Миграция*
```

## 💡 Советы

1. **Кратко** - максимум 15 строк
2. **Факты** - только то, что сделано
3. **Без кода** - только описание
4. **Регулярно** - после каждой задачи
5. **Ясно** - понятно через месяц
